# 实现与变更

用于实际写代码、修改现有功能、实现产品研发计划中的任务。

## 核心循环

1. 重新确认当前任务、范围和不做事项。
2. 先读相关代码、测试和现有模式，再编辑。
3. 能写测试时优先写聚焦测试；行为变化适合 TDD。
4. 小步实现，避免顺手重构。
5. 运行聚焦验证。
6. 行为正确后再做必要整理。
7. 根据影响范围决定是否运行更广的验证。

## TDD 使用条件

适合 TDD：

- 行为变化清楚。
- 项目已有可运行测试路径。
- 失败状态可以被稳定复现。

流程：

1. 写一个能表达目标行为的失败测试。
2. 确认失败原因符合预期。
3. 用最小实现让测试通过。
4. 必要时重构，并保持测试通过。

不适合 TDD 时，使用最可靠的替代证据：类型检查、lint、构建、局部复现、日志、浏览器验证、截图或人工检查清单。

## 既有项目安全变更

- 保持改动范围紧。
- 遵循本地架构、命名和文件边界。
- 不改无关行为。
- 不引入没有实际价值的抽象。
- 不覆盖用户已有改动。
- 多文件变更时，说明每个文件承担的角色。
