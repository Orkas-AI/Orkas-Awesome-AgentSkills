# Methodology Rules

## Selection Matrix

| User Scenario | Preferred Combination |
|---|---|
| Language exams such as IELTS/TOEFL | spaced review + Pareto + Pomodoro |
| Large-memory exams such as politics, law, medicine | heavy spaced review + Pareto |
| Programming, algorithms, and skill acquisition | Feynman output + Pomodoro |
| Math, physics, engineering, and conceptual subjects | Feynman output + Pareto |
| Fragmented time, commute, short breaks | Pomodoro + spaced review |
| Deep mastery rather than exam-only progress | Feynman output + light Pareto |

Use 1-3 methods in one plan. More than 3 usually adds cognitive load.

## Spaced Review

Use when the plan contains memory, vocabulary, formulas, concepts, procedures, error logs, or long-term retention.

Rules:

- Add reviews at expanding intervals such as next day, 3 days, 7 days, 14 days, and 30 days when the timeline allows.
- Label tasks as new learning, review, error review, or mixed retrieval.
- Pair reviews with active recall, not rereading.

## Pareto Prioritization

Use for exams, tight deadlines, sprint reviews, or "focus on the important parts" requests.

Rules:

- Identify high-frequency/high-weight topics first.
- Put high-value topics in the first 60% of the plan.
- Weight weak areas upward and strong areas downward.
- If time is short, set a drop line for low-frequency extras.

## Pomodoro Task Sizing

Use when the user has limited time, fragmented time, poor focus, procrastination, or many small sessions.

Rules:

- Keep single tasks at or below 45 minutes.
- Prefer 25-minute focus blocks for normal tasks.
- Add breaks between blocks when scheduling at hourly/minute granularity.
- If the user repeatedly fails to finish blocks, reduce the next block size instead of increasing pressure.

## Feynman Output

Use when the user wants to truly understand, apply skills, learn programming, or fix "I know it but cannot use it."

Rules:

- Add daily or weekly output tasks: explain, write, build, draw, teach back, record, or demo.
- Each week should include at least one output/checkpoint task when the plan is about mastery.
- Stage endings should include a demo, mock exam, teach-back, or self-explanation checkpoint.

## Time Budgeting

- Planned task time should usually be no more than 85-95% of the user's stated available time.
- For users with unstable schedules, keep a larger buffer.
- Do not let "professional-looking" plans become impossible to execute.
- If a plan cannot fit the deadline and budget, state the tradeoff clearly: extend the deadline, reduce the scope, or increase the time budget.
